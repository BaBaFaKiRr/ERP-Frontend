# User-based permissions — Phase 1 (inventory & design)

Status: **Phase 1 complete** — module/resource/action catalog for implementation in later phases.
See also: `invite_user.txt`, `Organisation_model.txt`, `FLOW.md` §3.

---

## 1. Product intent

- **Organization creator** → `organization_users.membership_role = owner` + module role `erp_admin` (full ERP access; bypasses `requireRole` gates when membership is owner/admin).
- **Invited employees** (HR → employee detail → **Invite to ERP**) → `membership_role = member` + **one** module role derived from department (`moduleRoleForDepartment` in `ERP-Backend/src/services/employee-invitation.ts`).
- **Target (Phase 2+)** → per-user, per-resource permissions (e.g. `accounts.sales_invoices.create`) stored in DB and enforced in API + UI, with **admin** able to customize grants per user beyond the department default.

Today the app uses **coarse module roles**, not granular permissions. Phase 1 defines the granular matrix to build next.

## 2. Access layers (today vs future)

| Layer | Storage | Effect today |
|-------|---------|--------------|
| **Membership** | `organization_users.membership_role` | `owner` / `admin` → all `requireRole` checks pass |
| **Module role** | `organization_user_roles.role` | One or more of: `erp_admin`, `accountant`, `store`, `production`, `sales` |
| **Legacy profile** | `public.users.role` | Mostly legacy; invite still inserts `sales` |
| **Granular permission** | *(not implemented)* | Phase 2: `user_permissions` or JSON on membership |

**Module role → department default (invite):**

| Employee department | Module role assigned |
|--------------------|----------------------|
| Sales, Impex, General Staff | `sales` |
| Production, Quality | `production` |
| Store, Dispatch | `store` |
| Accounts | `accountant` |
| Human Resources, Management | `erp_admin` |
| *(other / empty)* | `sales` |

## 3. Permission action vocabulary

Use these action codes consistently in Phase 2 schema and UI:

| Code | Meaning |
|------|---------|
| `fetch` | List / read / export PDF / preview |
| `create` | Insert new document or master record |
| `update` | Edit draft or mutable fields |
| `delete` | Hard or soft delete / cancel document |
| `approve` | Single-step approval (admin-only today for many docs) |
| `reject` | Reject pending approval |
| `request` | Submit for approval or raise a request queue item |
| `dispatch` | Physical dispatch / issue stock against invoice |
| `import` | Bulk import via data-import-export |
| `export` | Bulk export |
| `settings` | Org/module configuration (terms, invoice profiles, etc.) |

**Special rules (from FLOW.md — keep in enforcement):**

- Documents created by **admin / owner / erp_admin** often skip approval for the same action type.
- **Accounts / finance**: sensitive; default deny for `sales`, `store`, `production` unless explicitly granted.
- **Read-only** module access: grant only `fetch` (+ `export` if needed).

## 6. “Basic employee” default package (Phase 2 baseline)

When `membership_role = member` and no custom overrides:

1. Grant **one primary module role** from department (table in §2).
2. For that role, enable **fetch** on all resources in the same business area (e.g. sales sees sales orders + customers).
3. Enable **create/update** only on day-to-day operational resources listed in §5 “Suggested member” rows for that department.
4. Deny by default: **approve**, **reject**, **delete** (masters), **settings**, **hr.*** , **organization.invite**, **platform.admin**.
5. **Accountant** exception: full finance CRUD except approvals unless membership is admin/owner.

Admin UI (Phase 3) toggles individual `resource.action` flags per user.

## 7. Gaps vs target (for Phase 2 implementation)

| Gap | Detail |
|-----|--------|
| No `user_permissions` table | Only `organization_user_roles` + membership bypass |
| SO create unrestricted | POST `/sales-orders` has no `requireRole` — any member can create |
| Single module role per invite | `module_roles` array stored but only first role applied on accept |
| Frontend not gated | Sidebar shows all modules; no permission-aware nav |
| `public.users.role` | Legacy field; should not drive org authorization long-term |
